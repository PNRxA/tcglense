# Architecture ŌĆö annotated code map

The full, annotated file-by-file map of both apps (`api/` Rust backend, `web/` Vue
frontend), plus the test organization. `CLAUDE.md` is the always-loaded core and keeps
only a skeletal version of these trees; this file is the on-demand detail ŌĆö read it when
you need to know what a specific module does, where a concern lives, or how the pieces
wire together. Every path is cited the way the code cites itself; annotations are
carried near-verbatim from the audited source.

## Backend structure (`api/src/`)

test_support.rs    shared #[cfg(test)] fixtures: test_config(), a migrated in-memory SQLite DB, canonical card/set/user/holding rows (per-test tweaks via struct-update)

### Adding a backend feature

The step-by-step recipe (entity ŌåÆ migration ŌåÆ handler ŌåÆ route ŌåÆ wire types, including
the **frozen** `m20240101_0000NN` migration-naming rule and the two-place registration
in `migrator/mod.rs`) lives in `CLAUDE.md` ŌĆö always loaded, so it isn't duplicated
here. Handlers take `State(state): State<AppState>` for DB access and `AuthUser` to
require a signed-in user.

**Multi-TCG by design:** `cards`/`card_sets`/`ingest_state` carry a `game`
discriminator column; the catalog layer + routes are generic. Adding a TCG = add a
`Game` to `catalog::GAMES`, a provider module (like `scryfall/`), and one arm each
in `catalog::refresh_all` (live import) and `catalog::seed_all` (offline dummy seed).
On startup `tasks::start` spawns `catalog::refresh_all` in the background (gated by
`SYNC_ON_STARTUP`) so the server is up immediately, then re-runs it on a fixed
interval (`SYNC_INTERVAL_HOURS`, default 24 = daily) to pick up newer prices/sets;
the import streams the bulk file with bounded memory and **skips re-import when the
provider's `updated_at` is unchanged** (`ingest_state.source_updated_at`), so a tick
with no upstream change is cheap. When `SEED_DUMMY_DATA` is set, `tasks::start`
instead **awaits** `catalog::seed_all` (no network, no images) to populate a small
deterministic offline catalog and skips all syncing.

## Test organization

- **`api/src/security_tests/`** ŌĆö HTTP-level tests driving the *real* `build_router`
  in-process via `tower`'s `oneshot` (no TCP bind, no network). Each concern is its own
  module under `mod.rs`, over the shared `harness` (a capturing mailbox, an in-memory
  SQLite DB, and the full middleware stack): `caching`, `captcha`, `collection`,
  `collection_import`, `cors`, `email_verification`, `headers`, `login`, `mirror`, `pagination`,
  `password_reset`, `products`, `rate_limit`, `readiness`, `refresh`, `registration`, `request_body`,
  `search`, `sitemap`, `web_root`, `wishlist`, `wishlist_products` (`harness` itself is
  shared support, not a suite) ŌĆö ~20 suites. They assert the security-relevant behaviour no single-module unit
  test can see: end-to-end refresh rotation + reuse detection, generic login failures (no
  enumeration), the hardened refresh cookie on the wire, malformed-body status codes, the
  CORS + cache-header contracts, wishlistŌåöcollection independence, and that secret
  material never leaks into a response.
- **Search-compiler regression tests** live in `api/src/scryfall/search/tests.rs` ŌĆö they
  render each parsed query's WHERE clause to inlined SQLite **and** Postgres SQL to pin
  both the compiled shape and the deliberate per-backend divergences.
- **Opt-in / env-gated tests** (`#[ignore]`, run with `cargo test --locked -- --ignored`):
  the Postgres integration suite (`api/src/integration_pg.rs`, gated on
  `TCGLENSE_TEST_POSTGRES_URL`) and the Redis-limiter tests (in `api/src/ratelimit/backend.rs`,
  gated on `TCGLENSE_TEST_REDIS_URL`). The default suite + CI stay on in-memory SQLite;
  Postgres/Redis correctness is covered by these against docker
  (`deploy/docker-compose.yml`) and the `postgres-redis-tests` CI job.
- **Handler-level unit tests** live beside their handlers (e.g.
  `handlers/catalog/tests.rs`), and pure modules keep inline `#[cfg(test)]` tests
  (classifiers, mappers, folders ŌĆö `tcgcsv/classify.rs`, `scryfall/map.rs`,
  `mtgjson/model.rs`, ŌĆ”).

The card-catalog + sealed-product pages are **public** (no `requiresAuth`) and read
**public** endpoints, so they use `useQuery` from vue-query directly (not the
`useAuthedQuery` wrapper, which routes through `authFetch`). Card and product images are

On first navigation the guard calls `auth.tryRestore()` once: it hits
`/api/auth/refresh` (the httpOnly cookie is sent automatically) to mint an access

lives **only in memory** (no localStorage). The exported `authFetch` helper
transparently refreshes once on a 401 and retries, logging out if that still fails.
